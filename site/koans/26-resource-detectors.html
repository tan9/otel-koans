Knowing Where You Are

You named the service. But the environment has a name too — if you know how to ask.
span resource resource
service.name = orders
host.name = ip-10-0-3-47 Host detector
host.arch = amd64 Host detector
os.type = linux OS detector
cloud.provider = aws Cloud detector
cloud.region = us-east-1 Cloud detector
k8s.pod.name = orders-7b4f9c-xk2m K8s detector
k8s.namespace = production K8s detector
Most teams stop at service.name. But where is this service running? Which host? Which cloud region? Which Kubernetes pod?
Each detector queries the runtime environment at startup. No hardcoding. No config files. The attributes appear automatically.
Which detector discovered which attribute?
host.name, host.arch
→
Cloud detector
os.type
→
K8s detector
cloud.provider, cloud.region
→
Host detector
k8s.pod.name, k8s.namespace
→
OS detector

Why use resource detectors instead of setting attributes manually?

service.name is not discovered by a detector. Why?

service.name is yours to choose. Everything else — host, cloud, Kubernetes — resource detectors can discover automatically.

Name your service. Let the detectors handle the rest.
